# [SmartPlug]_3.1_모듈별 요구사항 명세서

# 1. Component 명세

## 1.1 공통

- 전제 조건
  - Hub 상에 ThinQ On 유사 수준의 기동어 인식 및 음성 제어 기능이 탑재되는 것을 전제함
- 플러그 - AP 간 Wi-Fi 연결
  - ThinQ 앱 연동 불가 : 부팅과 동시에 지정된 AP에 바로 연결되도록
- Hub의 기본 구조와 기능을 가능한 유지
  - 플러그는 remote mic+speaker 역할
  - Homey가 아닌 WebOS 영역으로 연동

# 2. 세부 사항

## 2.1 Summary

| 기동어 | Option1 | Option2 |
| --- | --- | --- |
| 인식 주체 | Hub | Plug |
| 컴포넌트 | ETWD, 서버 기동어 거절 기능 Disable | VIT |
| Pros. | 기존 기동어 처리 파이프라인 동일 경로 활용 | Hub 작업 단순화 |
| Cons. | 상대적으로 긴 데이터 경로(Plug → Hub → Plug)  서버기동어거절 기능 Off로 인한 잠재적 risk 대응 | 기존과 다른 다른 기동어 처리 방식 도입으로 인한 Hub 내 처리 시퀀스/  STM 수정 risk |

| 제어 명령어 | Option1 |
| --- | --- |
| 인식 주체 | Cloud (via Hub) |
| 컴포넌트 | ASR(MASA), TA, ThinQ |

| Cons. | (상동) |

| 인터콤 | Option1 | Option2 |
| --- | --- | --- |
| 명령어 식별 주체 | Plug | Cloud (via Hub) |
| 컴포넌트 | VIT(SI2) | ASR, NLU(PoC용 신규) |
| Pros. | 복수개의 사전 지정한 명령어를 이용, 구조/처리 경로 단순화 | ThinQ On의 명령어 처리 파이프라인과 유사 경로 향후 Cloud or Edge에서 인터콤 처리하기 위한 사전 검증 |
| Cons. | 명령어와 메시지 구분 불가, 명령어를 포함한 발화문 전체를 단순 전달 (Only Smart Plug PoC용) | ATOM 내 인터콤 PoC를 위한 NLU 신규 생성 작업 필요 (NLU 요구사항 구체화하여 인공연 전달 필요) |

## 2.2 플러그

- **기동어** : Edge(플러그 또는 허브)에서 처리  / ThinQ On의 서버 기동어 거절 기능 사용하지 않음 ( [Ref.](/main/spaces/~irene.hwang/pages/3645691939/25+_20250526_%EA%B8%B0%EB%8F%99%EC%96%B4+%EC%B2%98%EB%A6%AC+%EC%84%9C%EB%B2%84+%EA%B8%B0%EB%8F%99%EC%96%B4+%EA%B1%B0%EC%A0%88+disable+%EA%B4%80%EB%A0%A8) )\
  1안) Hub의 ETWD 사용\
  → 플러그는 음성 데이터를 수집하여 허브로 전달하고 허브로부터 결과/피드백을 수신하여 이를 재생하는 역할만 수행 : 플러그 구조 단순화(저성능 시스템 고려)\
      . 음성 데이터에 대해서는 단순 전처리( eg. 포맷 변환)와 재생에 필요한 후처리만 수행\
      . 서버 기동어 거절을 사용하지 않음에 따라 환경에 따른 인식률 저하가 있을 수 있으나 향후 허브 HW 구성이 변경될 경우 서버기동어거절 사용가능할 것으로 예상하고 현 상태 진행\
  2안) Plug 상에 NXP VIT 엔진 포팅, VIT에서 처리\
  → 플러그의 마이크를 통해 입력된 음성 데이터를 플러그 내부의 VIT 엔진(NXP)을 이용하여 기동어 식별. 허브나 클라우드로 전달하지 않음\
      이후 입력되는 명령문만 허브로 전달하여 처리. \
