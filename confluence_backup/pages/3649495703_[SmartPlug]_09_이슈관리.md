# 2. 이슈별 Root-Cause 분석

## **ISSUE\_01 — Plug HW mic/speaker bring-up 안됨 [B급]**

|  |  |
| --- | --- |
| 발생 시점 | 2025년 8월 5일 (시료 수령 직후) |
| 이슈 내용 | 내부 HW Task로부터 수령한 IoTPlug B/D에서 mic 입력 및 speaker 출력이 정상 동작하지 않음. 오디오 드라이버가 미완성 상태로 제공됨. |
| 근본 원인 | **[1차 원인]** 인수 기준(기본 동작 확인 5개 항목)을 제시하였으나, 공급자(BSP담당자)가 이를 충족하지 않은 상태로 HW를 제공함 **[근본]** 인수 기준이 존재했음에도 BSP 드라이버 완성도가 인수 기준 수준에 미달한 채 제공됨 → 사전 식별된 RISK\_04의 현실화 |
| 조치 내용 | BSP(Board Support Package) 전문가 자문을 통해 오디오 드라이버 최소 범위 수정. 일정 영향 없이 대응 완료. |
| 재발 방지 대책 | ① HW 인수 기준을 내부 협의 문서에 명시하고 미충족 시 재제공 요청 기준 마련 ② 실물 HW 투입 전 DevKit 환경에서 오디오 기능 사전 검증 완료를 Phase 1 진입 조건으로 설정 |
| RISK DB 갱신 | RISK\_04(Plug HW 음성/스피커 미지원) — 발생 확인 대응 이력을 반영하여 미충족 시 재제공 요청 기준 명시, DevKit 환경 사전 검증 완료를 Phase 1 진입 조건으로 설정(완화). 리스크 현실화 후 BSP 전문가 자문 투입(능동적 수용). |

## **ISSUE\_02 — Hub spec 부족: ATOM 기반 Voice Command 사용 불가 [B급]**

|  |  |
| --- | --- |
| 발생 시점 | 2025년 6월 (이슈 발생 식별) / 조치: 2025년 8월 |
| 이슈 내용 | On-Device AI Hub의 ATOM 기반 Voice Command 기능이 당시 Hub 빌드에서 미지원 상태임을 Phase 1-2 진입 후 확인. 해당 기능에 의존한 개발 계획 전면 재검토 필요. |
| 근본 원인 | **[1차 원인]** 타 과제 플랫폼(On-Device AI Hub)의 기능 스펙이 과제 시작 시점에 확인되지 않은 상태로 과제가 진행됨 **[근본]** 타 과제 플랫폼의 기능 스펙이 확인되지 않은 상태로 과제가 진행됨 |
| 조치 내용 | CCB를 통해 변경 수용. Google Gemini API 연동으로 대체 구현. 시연 시나리오는 동일하게 유지하되 AI 처리 경로 변경. 네트워크 지연에 대한 추가 검증 수행. |
| 재발 방지 대책 | ① 타 과제/플랫폼에 의존하는 기능 항목은 Phase 0 게이트 조건으로 기능 확인 완료를 명시 ② 플랫폼 의존 기능에 대한 대체 수단(Fallback)을 사전에 검토하여 리스크 대응 계획에 포함 |

## **ISSUE\_03 — Plug B/D 스테레오 오디오 미지원 [B급]**

|  |  |
| --- | --- |
| 발생 시점 | 2025년 8월 13일 |
| 이슈 내용 | IoTPlug B/D가 모노 오디오 출력만 지원. 시연 환경에서 스테레오 출력이 요구되었으나 HW 레벨에서 스테레오 미지원 확인. |
| 근본 원인 | 개발용 B/D(솔루션)가 모노 오디오로 구성되어 있었음 |
| 조치 내용 | 소프트웨어 처리로 회피: 모노 채널 오디오 데이터를 이중화하여 스테레오로 변환 후 재생. 시연 품질에 영향 없이 해결. |
| 재발 방지 대책 | HW 사양 제약으로 인한 오디오 출력 포맷 불일치는 SW 레벨(모노 데이터 이중화 등)에서 대응 가능함을 확인. 유사 제약 발생 시 SW 처리 방식을 우선 검토. |

## **ISSUE\_04 — Gemini API 연결 끊김 [A급]**

|  |  |
| --- | --- |
| 발생 시점 | 2025년 12월 4일 |
| 이슈 내용 | 특정 시점부터 Hub에서 Google Gemini API 연결이 차단됨. 서비스 가용성 문제로 AI 음성 명령 처리 전체가 불가 상태 발생. A급 이슈로 분류 (시연 핵심 기능 영향). |
| 근본 원인 | Google Gemini 무료(free) 플랜의 API 호출 한도가 정책 변경으로 축소됨 |
| 조치 내용 | Gemini 유료 플랜(Pay-as-you-go)으로 전환하여 API 연결 정상화. 이후 안정적 동작 확인. |
| 재발 방지 대책 | ① 시연/운영 환경에는 무료 플랜 미사용 원칙 수립 ② 외부 API 단일 의존 구조의 경우 fallback 방안을 설계 단계에서 검토 ③ 과제 예산 계획 시 외부 API 유료 사용 비용 항목 포함 |

## **ISSUE\_05 — Plug B/D Wi-Fi MAC Address 고정 불가 [A급]**

|  |  |
| --- | --- |
| 발생 시점 | 2025년 8월 2일 |
| 이슈 내용 | 개발용 B/D에 상용 MAC Address가 발급되지 않아 재부팅 시마다 MAC이 변경됨. Hub가 Plug를 고정 주소로 인식하지 못하는 연결 불안정 문제 발생. |
| 근본 원인 | 개발용 B/D에 상용 MAC Address가 발급되지 않음 (개발용 B/D의 제약) |
| 조치 내용 | 접속 아키텍처 변경: Soft AP를 Hub 대신 Plug에서 실행하고, Hub가 고정된 SSID를 기준으로 접속하도록 수정. 재부팅 후에도 안정적 연결 확인. |
| 재발 방지 대책 | 개발용 B/D에서 상용 MAC Address 미발급은 HW 환경 제약으로, 동일 환경에서의 근본적 방지는 어렵다. 유사 개발용 B/D를 사용하는 후속 과제에서는 초기 단계에 MAC 고정 가능 여부를 사전 파악하고, 필요 시 본 과제에서 검증된 Soft AP 역할 전환 방식을 적용한다. |

## **ISSUE\_09 — Gemini API 응답 Stuck [C급]**

|  |  |
| --- | --- |
| 발생 시점 | 2025년 11월 25일 |
| 이슈 내용 | Hub가 Gemini API에 요청을 보낸 후 응답이 오지 않고 무한 대기(stuck) 상태에 빠지는 현상이 간헐적으로 발생. 음성 명령 처리 흐름이 중단됨. |
| 근본 원인 | 외부 API 응답 지연/무응답 상황에 대한 timeout 처리 로직이 초기 구현에 없었음 |
| 조치 내용 | Timer 기반 timeout 구현: 일정 시간 내 응답이 없으면 재요청하도록 처리. 근본적인 무응답 원인은 미분석 상태로 추후 분석 예정으로 기록. |

## **ISSUE\_06 — 인터컴 기능 시연 범위 추가 [A급]**

|  |  |
| --- | --- |
| 발생 시점 | 2025년 8월 11일 (CR-01 변경 요청) |
| 이슈 내용 | 초기 합의된 시연 범위(음성 제어 중심)에 인터컴 기능이 추가로 요구됨. 팀원 2명이 투입된 지 2주 만에 발생한 A급 범위 변경. 개발 범위, 일정, 시연 시나리오 전면 재조정 필요. |
