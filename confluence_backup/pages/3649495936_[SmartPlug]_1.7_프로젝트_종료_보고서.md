# 5. Lesson Learned

| ID | 구분 | 내용 |
| --- | --- | --- |
| LL\_01 | 프로젝트 관리 방법론 | 시연 요구사항 및 시연 범위 변경으로 인해 Phase2 및 이후 일정에 큰 변동 발생  초기 요구사항 확정으로 발생 확률이 매우 낮을 것으로 예상하였으나, "고객"(상위관리자-iLab장)의 요청에 의해 예상보다 매우 큰 범위의 요구사항 변경이 발생하였고 이로 인해 CCB를 반복적으로 진행해야하는 상황이 발생함.  결과적으로 이와 같은 과제는 Agile 방식으로 진행하거나, Hybrid 방식으로 진행하는 것이 더 적절했을 것임. |
| LL\_02 | 점진적 가치 제시 | 인원 제약으로 Plug와 Hub가 처음부터 동시에 개발이 시작되지 못하고, 단계적으로 투입되어 하나의 기능이 구현 완료된 시점이라하더라도 사실상 고객에게 제공되는 각 단계별 가치를 설명하기가 쉽지 않은 문제.  기능 중심이 아니라, 사용자에게 단계적으로 가치를 제공한다는 관점에서 다시 고민해볼 필요가 있음. |
| LL\_03 | 이해관계자 소통 | SW 개발인원들이 처음 진행하는 분야의 업무로 경중과 상관없이 발생한 이슈를 혼자서 해결해보려고 붙들고 있는 경우가 자주 발생함.  아마도 이슈를 오픈한다해도 한정된 조직내 자원으로 인해 지원받기가 쉽지 않은 상황임을 감안하여 직접 해결해보려고 애를 쓰는 상황들이었을 텐데 조직 내부에서 도움을 받을 수 없으면 외부에서라도 도움을 받을 수 있다는 심리적 안정감을 제공하는 것이 필요하며, 이를 위해 어떤 것들이 보완되어야할지에 대해 후속 고민이 필요함 |
| LL\_04 | 이해관계자 소통 | 개발방법론 관점에서 Agile 도입을 이미 검토했었으나  현 프로젝트 팀 구성원이 Agile 방식의 경험이 전무하여, 사실상 도입이 불가능한 상황이었음  향후 유사한 상황에서 프로젝트 팀을 어떻게 Agile 방식으로 연착륙 시킬 것인지에 대해 고민이 필요함 |
| LL\_05 | 일정관리  - WBS 가시성 | CR-01/02 이후 신규 추가 항목에 대해 WBS를 세부로 나누지 않아 진척 수준의 가시성이 떨어지는 문제가 발생하였음. 일정이 촉박한 상황에서 업무 단위를 세밀하게 나누는 것에 대한 팀 내 심리적 저항이 있었으나, 진척 가시성 확보를 위한 WBS 세분화는 PM이 설득과 리더십을 통해 관철해야 할 영역임을 확인함. 후속 과제에서는 변경 발생 시에도 WBS 세분화 원칙을 사전에 합의하여 유지하는 것이 필요함. |
| LL\_06 | 일정관리  - 초기 계획 현실성 | 결과적으로 초기 계획 대비 큰 일정 영향 없이 과제가 완료되었으나, 도메인 지식·경험이 전무한 상태에서의 과도하게 낙관적인 초기 계획 수립은 위험함을 재확인함. 초기 단계에 적정 수준의 학습 시간을 보장하는 것이 후반부 리스크를 감소시키는 중요한 요소로 작용함. LL\_08의 사전 PC기반 검증 계획과 동일한 맥락으로, 초기 투자가 후반 변동 대응력을 높인다는 교훈을 다시 한 번 확인함. |
| LL\_07 | 리스크 관리 | LL\_01과 관련하여 "요구사항" 뒤의 "고객"이 "어떤 유형의 사람"인가를 파악하는 것 역시 리스크 관리 포인트가 되어야 함 |
| LL\_08 | 리스크 대응 계획 수립 | Plug HW 확보 일정 지연이 발생하게 될 경우의 리스크를 최소화하기 위해 시료 확보 전 PC기반으로 성능 및 기능 검증을 할 수 있는 범위 내에서 최대한 진행하도록 계획을 수립한 것이 후반부 큰 변동이 발생했음에도 실제 개발 일정이 지연되지 않을 수 있었던 중요한 결정이었음.  리스크가 무엇인지 단순 나열하는데서 나아가, 현실적으로 어떻게 위험수위를 낮출 것인가를 직접적으로 고민해본 좋은 계기였음. |
| LL\_09 | 외부 의존성 관리 | 타 과제/플랫폼에 의존하는 기능은 과제 착수 전에 스펙 확인을 완료해야 한다. 확인되지 않은 상태로 개발 계획을 수립하면 플랫폼 지원 여부에 따라 개발 범위 전체가 변경될 위험이 있으며, 이 경우 CCB를 통한 대응이 불가피해진다. (ISSUE\_02 사례 참조) |
| LL\_10 | 외부 서비스 의존성 관리 | 외부 API 무료 플랜은 정책 변경으로 한도가 축소되거나 차단될 수 있다. 시연·운영 환경에는 무료 플랜 사용을 지양하고, 과제 예산 계획 시 유료 API 비용 항목을 포함해야 한다. 외부 API 단일 의존 구조의 경우 fallback 방안도 사전에 검토할 것. (ISSUE\_04 사례 참조) |
| LL\_11 | 이해관계자 관리 | 최상위 이해관계자(or 이해관계자의 권력 수준이 높을수록)에 의한 범위 변경은 기존 변경 통제 프로세스가 있어도 현실적으로 통제하기 어렵다. 이를 사전에 현실적 리스크로 인식하고, 일정·자원에 여유 버퍼를 반영하는 것이 실질적 대응이다. (ISSUE\_06 사례 참조)  즉, 이해관계자 참여 전략의 실효성은 참여 빈도와 활동 수행 여부만으로 측정될 수 없으며 이해관계자에 의한 범위 변경 리스크를 사전에 통제하는 구조적 장치가 병행되어야 함. |
