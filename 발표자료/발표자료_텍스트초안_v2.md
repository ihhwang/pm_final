**[좌측: 참여 활동 요약]**

| 채널 | 대상 | 주기 | 실적 |
|---|---|---|---|
| 주간 팀미팅 | TL, 개발 | 매주 | 31회 / 참석률 100% |

| 게이트 리뷰 | 전체 이해관계자 | 최소 2회 | 3회 진행 |

**[우측: CR 발생과 전략 판단]**

CR-01/02는 정기 채널 외 경로(iLab장 직접 지시)로 발생 → 참여 전략만으로는 구조적 차단 불가

**검토한 대응 옵션:**
- 참여 채널·빈도 조정 → **효용 없음으로 판단 및 폐기**
  - 최상위 이해관계자의 권한 행사를 채널 조정으로 차단할 수 없음
- **선택: 일정·자원 버퍼 흡수 대응**
  - CR 이전 3개 WBS 선행 완료로 대응 여력 확보

## S8. PM 없이 된 것 vs 안 된 것 (1.5분)

**[제목]** 공식 PM 없이 — 무엇이 작동했고 무엇이 작동하지 않았는가

**[좌측: 된 것 — 수치 근거]**

| 관리 영역 | 수행 수준 | 근거 |
|---|---|---|
| 요구사항 정의 및 승인 | 44개 요구사항, 이해관계자 공식 승인 | 3.3 베이스라인 |
| 변경 영향도 분석 | CR 발생 시 정량 분석 + CCB 수행 | 6.1 §4 |
| 리스크 → 이슈 전환 관리 | RISK 7건 전수 추적, 이슈 전환 후 전원 Closed | 9.1 / 9.4 |

| 교훈 도출 | LL 11건 문서화 | 1.7 |

**[우측: 안 된 것 — 원인까지]**

| 관리 영역 | 수준 | 원인 |
|---|---|---|
| 추적 체계 시각화 | 매트릭스는 있으나 한눈에 파악 어려움 | 도구 부재 (JIRA 등 미사용) |
| KPI 임계치 사전 명문화 | 판단은 했으나 기준이 문서화 안 됨 | PM 역할 부재로 관리 체계 설계 생략 |
| 변경통제 사전 차단 | CCB 수행했으나 절차 외 요청 발생 | 최상위 이해관계자 권한 구조적 한계 |
| 정보 배포 체계 | 공유는 됐으나 배포 주기·권한 미문서화 | 형식적 소통 계획 수립 부재 |

## S9. 재시작한다면: 보완 계획 3가지 (1.5분)

**[제목]** 다시 시작한다면 PM으로서 이렇게 하겠습니다

**[보완 1 — 심사위원1·2 코멘트 직접 대응]**
**KPI 임계치 + 판단 기준 초기 명문화**
- 착수 시 KPI별 임계치·판단 규칙·조치 책임자를 단일 표로 확정
- 측정 주기와 보고 트리거를 관리 계획서에 포함
- 현재: 판단은 했으나 기준이 사후에만 설명 가능 → 향후: 기준이 사전에 고정됨

**[보완 2 — 심사위원3 코멘트 직접 대응]**
**REQ → WBS → 검증 추적 체계 시각화**
- 요구사항 추적 매트릭스에 상태값(계획/진행/완료/변경) 컬럼 추가
- 대시보드에서 요구사항별 현재 상태를 한 화면에서 확인 가능하도록 구성
- 현재: 표는 있으나 상태 파악에 문서 탐색 필요 → 향후: 단일 뷰에서 즉시 확인

**[보완 3 — 전체 공통]**
**초기 단계에서 변경 리스크 버퍼 명시적 반영**
- 최상위 이해관계자 변경 요청 가능성을 리스크로 사전 등록
- 일정·자원에 명시적 버퍼(예: 전체 일정의 10~15%) 확보
- 현재: 우연히 선행 완료로 대응 여력 확보 → 향후: 계획 단계에서 의도적으로 설계
